## 1. プロダクト・コンセプト（仮確定）

### 1.1 一言でいうと
> スマホで「ながら世話」できる、犬・猫を育てる**放置型ペット育成アプリ**。
> アプリを閉じている間も時間が進み、戻ってくると「お腹が空いた」「成長した」が待っている。

### 1.2 「放置型」と「たまごっち」の定義（本プロジェクトでの意味）

| 要素 | たまごっち的（世話の手触り） | 放置型（idle）的（時間が報酬を生む） |
|---|---|---|
| ステータス | 空腹・機嫌・清潔・体力が**時間で減る** | 閉じている間も裏で進行（オフライン計算） |
| プレイヤー操作 | ごはん・遊ぶ・掃除・なでる | 自動でコイン/経験値が貯まる |
| 失敗条件 | 放置しすぎると不調・家出（緩め） | （任意）通貨で世話を自動化＝放置強化 |
| リテンション | プッシュ通知で呼び戻す | 「久しぶりに来たら大量報酬」体験 |

> **設計の肝**: 「世話の能動性（たまごっち）」と「放置の受動報酬（idle）」のバランス。
> 初期方針 = **やさしい放置**（死亡・全損なし、最悪でも体調ダウンまで）。ライト層の離脱を防ぐ。

### 1.3 想定ユーザー（仮）
- メイン: スマホで隙間時間に癒やしを求めるライト層（ペット好き／10〜30代）
- 1日の想定接触: 2〜5回 × 1回30秒〜2分

### 1.4 コアループ（1セッション）
```
通知で起動 → 溜まった報酬を回収 → 減ったステータスを世話で回復
        → 通貨で成長/着せ替え/環境アップグレード → アプリを閉じる（裏で進行継続）
```

## 2. スコープ定義（MVPでやること / やらないこと）

### 2.1 MVP（v0.1）でやる ✅
- 犬 or 猫を1匹選んで飼う（種類は2〜3）
- 4ステータス（空腹・機嫌・清潔・体力）の時間減衰
- 世話アクション4種（ごはん・遊ぶ・掃除・なでる）
- 放置報酬（コイン）の自動蓄積＋オフライン分の回収
- 成長段階（赤ちゃん→子→成体の3段階）
- ローカル保存（端末内、ログイン不要）
- プッシュ通知1種（「お腹が空いたよ」）

### 2.2 MVPでやらない ❌（v0.2以降）
- 多頭飼い／繁殖
- 課金・広告（収益化）
- クラウドセーブ・アカウント連携
- ランキング・フレンド・SNS共有
- ガチャ／着せ替えの大量バリエーション
- ミニゲーム

> **理由**: 放置ゲームの肝（オフライン進行＋通知でのリテンション）が成立するかを最小構成で検証する。
> 収益化・ソーシャルは「面白さ」が確認できてから載せる（過剰投資の回避）。

## 3. 機能要件 / 非機能要件

### 3.1 機能要件（MVP）
| ID | 要件 | 完了条件（受入基準） |
|---|---|---|
| F-01 | ペット選択 | 初回起動で種類を選び、以後固定で表示される |
| F-02 | ステータス減衰 | 実時間に比例して各値が減る（後述レート） |
| F-03 | 世話アクション | 各アクションで対応ステータスが回復しアニメ反応する |
| F-04 | 放置報酬 | 経過時間に応じコインが貯まり、復帰時に回収できる |
| F-05 | オフライン進行 | アプリ終了→再起動でも経過時間が正しく反映される |
| F-06 | 成長 | 累積世話/経過でステージが上がり見た目が変わる |
| F-07 | 永続化 | 端末を再起動してもデータが残る |
| F-08 | 通知 | 空腹が閾値を下回ると通知が届く |

### 3.2 非機能要件
- 起動〜操作可能まで **2秒以内**（ライト層は待たない）
- オフライン進行計算は **誤差なく決定論的**（同じ入力→同じ結果）
- データ破損時も**初期化で復帰**できる（クラッシュさせない）
- 端末ローカルで完結（MVPは個人情報を持たない＝プライバシーリスク最小）

## 4. ゲームデザイン仕様（数値で書く）

> ※ 数値は初期値。プレイテストで必ず再調整する「仮の杭」。

### 4.1 ステータス減衰レート（例）
| ステータス | 範囲 | 減衰 | 補足 |
|---|---|---|---|
| 空腹(hunger) | 0–100 | -100 / 8h | 0で機嫌・体力が追加減少 |
| 機嫌(mood) | 0–100 | -100 / 12h | なでる/遊ぶで回復 |
| 清潔(clean) | 0–100 | -100 / 16h | 放置で機嫌に微影響 |
| 体力(energy) | 0–100 | 睡眠で回復 | 遊ぶと消費 |

### 4.2 オフライン進行のロジック（最重要）
```
復帰時の処理:
  now      = 現在時刻
  last     = 最終保存時刻（保存済み）
  elapsed  = clamp(now - last, 0, MAX_OFFLINE)   // 例: MAX_OFFLINE = 24h で上限
  各status = clamp(status - rate * elapsed, 0, 100)
  coin     = coin + coinRate * elapsed            // 放置報酬
  画面に「○時間ぶり！コイン+N」を提示
```
- **MAX_OFFLINE で上限**を設ける（無限蓄積で放置ゲーが壊れるのを防ぐ）
- 端末時計の改ざん対策はMVPでは**簡易**（巻き戻り検知で進行を0に）。本格対策はサーバー時刻同期＝v0.2以降。

### 4.3 経済（コイン）
- 用途: ごはん購入 / 成長促進 / （v0.2）着せ替え
- 入手: 放置報酬 ＋ 世話アクションのボーナス
- 初期は**インフレ防止より「気持ちよさ」優先**（数字が増える快感）

### 4.4 成長ステージ
- Stage1 赤ちゃん（0〜）→ Stage2 子（世話N回 or 経過T）→ Stage3 成体
- 各ステージで見た目（スプライト）と できる遊びが増える

## 5. 技術選定（推奨と理由）

> ★プラットフォームは承認Gate項目。以下は**部長推奨案**。

### 5.1 推奨スタック（案A: Web/PWA優先）
| レイヤ | 採用 | 理由 |
|---|---|---|
| 形態 | **PWA（モバイルWeb）** | ストア審査なしで即リリース→検証が速い。クロスプラットフォーム |
| 言語/UI | TypeScript + React + Vite | 標準的で人材・情報が豊富。型でバグ予防 |
| 状態管理 | Zustand | 軽量。放置ゲームの単純な状態に最適（Redux過剰） |
| 永続化 | IndexedDB（idb-keyval） / localStorage | オフライン保存。容量と将来性でIndexedDB寄り |
| アニメ | CSS / Framer Motion / Lottie | まずCSSスプライト、リッチ化でLottie |
| 通知 | Web Push + Service Worker | PWAでの呼び戻し |
| ホスティング | Vercel / Netlify | 無料枠・CI連携・即デプロイ |
| バックエンド | **MVPはなし** → v0.2でSupabase | 認証/クラウドセーブ/ランキングを段階導入 |

**案A採用理由**: リリース速度と検証コストが最小。放置ゲームは「通知で戻れるか」が命で、PWA通知で十分検証可能。

### 5.2 代替案（比較）
| 案 | 長所 | 短所 | 向くケース |
|---|---|---|---|
| 案A PWA | 速い・安い・審査なし | iOSのWeb Push制約・ネイティブ感弱め | まず検証したい（推奨） |
| 案B React Native/Expo | 1コードでネイティブ通知・ストア配信 | ビルド/審査の手間 | 通知リテンション本気 |
| 案C Flutter | 描画リッチ・高性能 | Dart学習・人材 | アニメ凝りたい |
| 案D Unity | 2D表現最強・アセット豊富 | 重い・Web配信不利 | ゲーム表現が主役 |

> **部長の見立て**: 「面白いか」を最短で確かめる目的なら**案A→当たれば案B移行**が筋。
> OSSライセンス: React(MIT)/Vite(MIT)/Zustand(MIT)/idb-keyval(MIT)/Framer Motion(MIT) — いずれも**商用可・問題なし**。Lottie(MIT) / アニメ素材は別途ライセンス確認要。

## 6. データモデル（MVP）

```ts
// すべて immutable に更新（{...state, ...} で新規生成。直接 mutate しない）
type PetKind = 'dog' | 'cat';
type Stage = 1 | 2 | 3;

interface PetState {
  kind: PetKind;
  name: string;
  stage: Stage;
  hunger: number;   // 0-100
  mood: number;     // 0-100
  clean: number;    // 0-100
  energy: number;   // 0-100
  coin: number;
  careCount: number;     // 成長判定用
  lastSavedAt: number;   // epoch ms（オフライン計算の基点）
  version: number;       // セーブマイグレーション用
}
```
- 入力（名前など）は **Zod でバリデーション**
- セーブは `version` を持たせ、将来のスキーマ変更でマイグレーション可能に

## 7. UI/UX デザイン手順

### 7.1 画面（MVP）
1. **オンボーディング**: 犬/猫選択 → 名付け
2. **ホーム（メイン）**: ペット大写し＋4ステータスバー＋世話ボタン4つ＋コイン表示
3. **復帰モーダル**: 「○時間ぶり！ コイン+N、お腹が空いてるよ」
4. **設定**: 通知ON/OFF・データリセット

### 7.2 デザイン作業の進め方
| 手順 | 成果物 | ツール |
|---|---|---|
| D-1 ムードボード | トーン&マナー（かわいい/手描き/ドット 等の方向決め） | Figma / 画像収集 |
| D-2 ワイヤーフレーム | 画面遷移と要素配置（白黒） | Figma |
| D-3 ビジュアルデザイン | 配色・タイポ・アイコン | Figma + `design-consultation` skill 活用可 |
| D-4 ペットアート | 種類×3ステージ×状態（通常/喜/空腹）スプライト | 自作 / 外注 / 生成 |
| D-5 プロトタイプ | 触れるFigmaプロト or HTMLモック | Figma / frontend-design skill |

> **アート方針の決定が必要**（承認Gate）: 自作ドット / 外注イラスト / 生成AI素材 のいずれか。コストと世界観に直結。

### 7.3 UXトーン
- 失敗に厳しくしない（家出はするが死なない）。罪悪感より「会いたくなる」設計。
- マイクロインタラクション（なでると揺れる/喜ぶ）で手触りを作る。

## 9. テスト計画

| レベル | 対象 | 手段 |
|---|---|---|
| 単体 | オフライン進行計算・減衰・コイン蓄積 | Vitest（**決定論なので最優先で厚く**） |
| 単体 | セーブ/ロード・マイグレーション | Vitest（破損データ投入も） |
| 結合 | 世話→ステータス→UI反映 | Testing Library |
| E2E | 選択→世話→閉じる→（時計進めて）→復帰報酬 | Playwright（webapp-testing skill） |
| 手動 | 通知が実機で届くか | iOS/Android実機 |

- カバレッジ目標: **コアロジック80%以上**（UIは緩め）
- 時間依存はテストで `now` を注入できる設計に（`Date.now()` 直呼び禁止＝モック可能に）
